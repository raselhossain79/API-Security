# OAuth 2.0 Flows Explained as Raw HTTP

## Purpose of This File

Every attack in this series depends on understanding what a *correct* OAuth flow looks like on the wire. You cannot spot a manipulated `redirect_uri`, a missing `state`, or a scope-upgrade attempt unless you first know exactly which parameter belongs in which request, and why. This file has no attacks in it. It is the baseline.

## The Three Roles

- **Client application** — the app that wants access to your data (e.g., a blog that lets you "log in with Google").
- **Resource owner** — you, the user.
- **Authorization server / OAuth provider** — the service that authenticates you and issues tokens (e.g., Google, GitHub, an internal identity provider like Okta or Keycloak).

A fourth role, the **resource server**, is where the actual protected data lives (e.g., `/api/userinfo`). In many deployments the authorization server and resource server are the same host, but they don't have to be.

## Flow 1: Authorization Code Grant

**When it's used:** Server-side web applications (classic client-server apps) where a backend can securely hold a `client_secret`. This is the most common flow for "Login with X" buttons and the one most attacks in this series target.

**Why it exists:** It keeps the access token off the front channel (the browser). The browser only ever sees a short-lived, single-use `code`. The actual token exchange happens server-to-server over a "back channel" that an attacker on the network or in the browser cannot observe.

### Step 1 — Authorization Request

The client redirects the user's browser to the authorization server:

```
GET /auth?response_type=code&client_id=app123&redirect_uri=https://client-app.com/callback&scope=openid%20profile%20email&state=af0ifjsldkj HTTP/1.1
Host: auth.oauth-provider.com
```

Line-by-line:
- `response_type=code` — tells the authorization server which grant type to run. `code` means "give me an authorization code, not a token, at the end of this." This single parameter is why an attacker who can rewrite it to `token` may force the server into implicit-flow behavior if the server doesn't strictly validate which response types a client is registered for.
- `client_id=app123` — identifies which registered application is making the request. The authorization server uses this to look up the application's registered redirect URIs, allowed scopes, and allowed response types. Everything else in this request is validated *against* the configuration tied to this `client_id`.
- `redirect_uri=https://client-app.com/callback` — where the browser should be sent after the user approves access. This is the single most attacked parameter in OAuth, covered in depth in file 2.
- `scope=openid profile email` — the permissions being requested. Space-separated, URL-encoded as `%20`. The user will be shown a consent screen listing these. Manipulating this list is covered in file 4.
- `state=af0ifjsldkj` — an opaque value generated by the client, tied to the user's current session, that the authorization server must echo back unchanged. Its entire job is CSRF protection for the client's callback endpoint. Covered in depth in file 3.

### Step 2 — User Authenticates and Consents

The authorization server shows a login form (if not already logged in) and a consent screen ("client-app.com wants to access your profile and email"). This step produces no interesting raw HTTP for our purposes beyond the `POST` of credentials to the authorization server itself — that traffic belongs to the authorization server's own authentication system, not to OAuth.

### Step 3 — Authorization Response (Redirect Back)

```
HTTP/1.1 302 Found
Location: https://client-app.com/callback?code=SplxlOBeZQQYbYS6WxSbIA&state=af0ifjsldkj
```

Line-by-line:
- `code=SplxlOBeZQQYbYS6WxSbIA` — a single-use, short-lived (often 30–60 seconds) authorization code. It has no value on its own — it must be redeemed at the token endpoint together with the client's secret. This is what makes code interception (file 2) less immediately catastrophic than implicit-flow token interception, but still exploitable if the receiving client doesn't also verify the redirect_uri at exchange time.
- `state=af0ifjsldkj` — echoed back exactly as sent. The client must compare this to the value it stored before redirecting the user in Step 1. If they don't match, the client must reject the response outright.

### Step 4 — Token Request (Back Channel, Server-to-Server)

```
POST /token HTTP/1.1
Host: auth.oauth-provider.com
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&code=SplxlOBeZQQYbYS6WxSbIA&redirect_uri=https://client-app.com/callback&client_id=app123&client_secret=cKhx8dGF3EqTaOtHqzJHKZ8G
```

Line-by-line:
- `grant_type=authorization_code` — tells the token endpoint which exchange logic to run. This is a separate check from `response_type` in Step 1; a server that validates one but not the other opens room for grant-type confusion attacks.
- `code=...` — the code from Step 3. The server looks this up, confirms it hasn't been used before (single-use), confirms it hasn't expired, and confirms it was issued to this exact `client_id`.
- `redirect_uri=...` — sent again here. A properly implemented authorization server compares this value to the one supplied in Step 1 for this same code, and rejects the exchange on mismatch. This second check is what neutralizes a large class of redirect_uri attacks — an attacker can trick a victim's browser into sending a code to an attacker-controlled callback, but redeeming that code still requires the attacker to supply the *original* redirect_uri here, which is fine for the attacker (they already have the code) but is the control point many implementations get wrong by skipping this check entirely.
- `client_id` and `client_secret` — proves the caller is the legitimate registered client, not just anyone who intercepted a code. This exchange happens server-to-server, so a browser-based attacker cannot normally supply their own valid `client_secret` — unless the attacker controls their own registered malicious client application, which is the basis of the scope-upgrade attack in file 4.

### Step 5 — Token Response

```
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: no-store

{
  "access_token": "2YotnFZFEjr1zCsicMWpAA",
  "token_type": "Bearer",
  "expires_in": 3600,
  "refresh_token": "tGzv3JOkF0XG5Qx2TlKWIA",
  "scope": "openid profile email"
}
```

- `access_token` — used by the client to call the resource server's API (e.g., `/userinfo`) on the user's behalf.
- `refresh_token` — used later to get a new `access_token` without re-prompting the user. Never exposed to the browser in this flow.
- `Cache-Control: no-store` — a real server should send this so intermediary caches and browser history don't retain the token response. Its *absence* is itself a real-world finding worth noting in a report.

## Flow 2: Client Credentials Grant

**When it's used:** Machine-to-machine (M2M) communication with no end user involved at all — a backend service calling another backend service's API using its own identity. Think microservice-to-microservice auth, CI/CD pipelines calling an internal API, or a scheduled job pulling data from a partner's API.

**Why it exists:** There is no "user" to redirect or consent. The client authenticates as itself.

```
POST /token HTTP/1.1
Host: auth.oauth-provider.com
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&client_id=service-a&client_secret=Ry8fH2kLpQ9vXeZ3&scope=reports:read
```

- `grant_type=client_credentials` — single request, no redirect, no browser involvement at all. This is the entire flow.
- `client_id` / `client_secret` — this pair *is* the identity being asserted. There's no separate user credential.
- `scope=reports:read` — the service is requesting only the permission it needs. From an attack perspective, over-broad scopes granted to service accounts (e.g., a reporting service that can also `reports:write` or `users:delete`) are a common real-world excessive-privilege finding, since there's no user-facing consent screen to catch it.

Response is a plain access token, same shape as Step 5 above, minus any `refresh_token` (since the client can just request a new token any time using its own credentials).

**Real-world note:** Because this flow has no browser leg at all, none of the browser-based attacks in this series (redirect_uri manipulation, state/CSRF, Referer leakage) apply to it. If you find `client_credentials` in the wild, your attack surface shifts to credential exposure (leaked `client_secret` in mobile app binaries, JS bundles, CI logs, public GitHub repos) rather than flow manipulation.

## Flow 3: Device Authorization Grant ("Device Flow")

**When it's used:** Devices with no browser or a very limited input method — smart TVs, CLI tools, IoT devices. You've seen this as "Go to example.com/activate and enter code ABCD-1234."

### Step 1 — Device Requests a Code

```
POST /device/code HTTP/1.1
Host: auth.oauth-provider.com
Content-Type: application/x-www-form-urlencoded

client_id=tv-app-42&scope=openid profile
```

### Step 2 — Server Issues Two Codes

```
HTTP/1.1 200 OK
Content-Type: application/json

{
  "device_code": "GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS",
  "user_code": "WDJB-MJHT",
  "verification_uri": "https://auth.oauth-provider.com/device",
  "expires_in": 1800,
  "interval": 5
}
```

- `device_code` — long, unguessable, stays on the device. Never shown to the user.
- `user_code` — short, human-typeable, shown on the device's screen. This is what the user enters on a *separate* trusted device (phone/laptop).
- `verification_uri` — where the user goes to enter `user_code`.
- `interval` — how often (seconds) the device is allowed to poll the token endpoint while waiting.

### Step 3 — Device Polls for a Token

```
POST /token HTTP/1.1
Host: auth.oauth-provider.com
Content-Type: application/x-www-form-urlencoded

grant_type=urn:ietf:params:oauth:grant-type:device_code&device_code=GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS&client_id=tv-app-42
```

While the user hasn't yet entered the code on the separate device, the server keeps returning:

```
HTTP/1.1 400 Bad Request
{"error": "authorization_pending"}
```

Once the user enters `user_code` and approves on the verification page, the next poll returns a normal token response as in Flow 1, Step 5.

**Attack-relevant note:** The real-world weakness in device flow isn't a parameter to tamper with — it's **phishing the user_code itself**. An attacker starts their own device-flow session, gets a legitimate `user_code`, and social-engineers the victim into entering *that* code on the real verification page (e.g., disguised as tech support, a "verify your account" prompt, or embedded in a phishing page that mimics a legitimate login). The victim thinks they're logging into their own device; they're actually authorizing the attacker's session. This is why device flow UX guidance stresses short `user_code` lifetimes, rate-limiting entry attempts, and showing enough context (application name, requested scope) on the verification page for the user to notice something is wrong.

## Flow 4: Implicit Grant

**When it's used:** Historically used for single-page applications (SPAs) and JavaScript apps that couldn't securely hold a `client_secret`. **It is now deprecated by the OAuth 2.0 Security Best Current Practice (BCP) in favor of Authorization Code + PKCE**, but it remains extremely common in legacy systems and is worth understanding in depth because it's still frequently found in real-world assessments and is the subject of several PortSwigger labs.

### Step 1 — Authorization Request

```
GET /auth?response_type=token&client_id=spa-app&redirect_uri=https://client-app.com/callback&scope=openid%20profile&state=xyzABC123 HTTP/1.1
Host: auth.oauth-provider.com
```

Only difference from the code flow: `response_type=token` instead of `response_type=code`. This one word changes everything about what comes back.

### Step 2 — Token Delivered Directly via URL Fragment

```
HTTP/1.1 302 Found
Location: https://client-app.com/callback#access_token=2YotnFZFEjr1zCsicMWpAA&token_type=Bearer&expires_in=3600&state=xyzABC123
```

Line-by-line:
- The token is appended after a `#`, not a `?`. This is deliberate: URL fragments are **never sent to the server** by the browser on the initial page load — only JavaScript on the page can read `location.hash`. This was meant to keep the token out of server access logs.
- `access_token` arrives directly in the browser with **no client secret and no back-channel exchange at all**. There is no equivalent of Flow 1's Step 4. The browser has the live, usable token the instant the redirect lands.

### Step 3 — Client-Side JavaScript Reads the Fragment and Establishes a Session

Because the SPA needs to persist the login across page reloads, it typically does this (pseudocode of what fires on page load):

```
POST /authenticate HTTP/1.1
Host: client-app.com
Content-Type: application/json

{"access_token":"2YotnFZFEjr1zCsicMWpAA","email":"user@example.com","username":"user123"}
```

This is the client's own backend saying "trust this data because it came with a valid-looking token." **This is the exact request targeted by the classic implicit-flow authentication bypass**: if the backend does not independently verify that the `access_token` actually belongs to the `email`/`username` in the same request (e.g., by calling the OAuth provider's `/userinfo` endpoint itself with that token), an attacker can simply replay this `POST` with a victim's email address and their *own* valid token, and the backend will log them in as the victim. The token proves the attacker authenticated as *someone*; it proves nothing about *which* identity fields in this specific request are genuine unless the server checks.

**Why the fragment token is dangerous beyond that:**
- It lands directly in browser history.
- It's visible in the URL bar during the redirect.
- If any script on the page (analytics, ads, a vulnerable third-party widget) reads `location.href` or `document.referrer` carelessly, or if the page navigates again and a `Referer` header carries the full URL, the token can leak off-domain. This is expanded in file 4.

## Quick Comparison Table

| Flow | Browser involved | Client secret needed | Token exposed to browser | Typical use case |
|---|---|---|---|---|
| Authorization Code | Yes | Yes (confidential client) | No (only a code) | Server-side web apps |
| Authorization Code + PKCE | Yes | No | No (only a code) | SPAs, mobile apps |
| Client Credentials | No | Yes | N/A | Service-to-service |
| Device Code | Secondary device only | Sometimes | No | TVs, CLI, IoT |
| Implicit | Yes | No | **Yes, directly** | Legacy SPAs (deprecated) |
