# PKCE Bypass: When the Fix Itself Is Misconfigured

## Why PKCE Exists — Tying Back to File 2

PKCE (Proof Key for Code Exchange, RFC 7636, pronounced "pixy") was designed to close exactly the hole File 2 spent an entire file exploiting: an authorization code, once obtained by any means — interception, redirect_uri manipulation, Referer leakage, a shared device — being redeemable by **anyone** who has it, with no way for the token endpoint to verify that the party redeeming the code is the same party that originally requested it.

PKCE was originally built for **public clients** (mobile apps, SPAs) that can't hold a `client_secret` confidentially, since a compiled mobile app or client-side JavaScript bundle can always be reverse-engineered to extract any embedded secret. But it's now recommended for **confidential clients too** (OAuth 2.1 makes it effectively universal) precisely because it adds a second, independent layer of protection against code interception on top of redirect_uri validation and client secrets — defense in depth against exactly the attack chain in File 2, Section 1.

Prerequisite: File 1's Authorization Code flow walkthrough and File 2's explanation of why stealing a code is often sufficient for account takeover without ever needing the `client_secret`. This file assumes you understand that baseline attack before covering how PKCE is meant to stop it and where that stopping mechanism breaks down.

## 1. The Correct Mechanism

**Step 1 — The client generates a code verifier**

Before redirecting the user, the client generates a high-entropy random string, kept **only in the client's memory** — never sent anywhere yet:

```
code_verifier = "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk"
```

**Step 2 — The client derives a code challenge from it**

```
code_challenge = BASE64URL-ENCODE(SHA256(code_verifier))
```

This is a one-way transformation — the challenge can be computed from the verifier, but the verifier cannot be recovered from the challenge. This directional property is the entire security value of PKCE, so it's worth being explicit about why: anyone who intercepts the `code_challenge` (which travels in the open, in the authorization request) learns nothing that lets them compute a valid `code_verifier`.

**Step 3 — The authorization request includes the challenge, not the verifier**

```
GET /auth?client_id=a1b2c3&redirect_uri=https://client-app.com/callback&response_type=code&scope=openid%20profile&state=xyz789&code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM&code_challenge_method=S256 HTTP/1.1
Host: oauth-authorization-server.com
```

Breaking down the two new parameters:
- `code_challenge` — the derived value from Step 2. Safe to expose publicly, since it can't be reversed.
- `code_challenge_method=S256` — tells the authorization server which transformation was used (SHA-256 in this case) so it knows how to verify the eventual verifier against this challenge.

The authorization server stores this challenge server-side, associated with the authorization code it's about to issue.

**Step 4 — The authorization code is issued as normal**

```
HTTP/1.1 302 Found
Location: https://client-app.com/callback?code=SplxlOBeZQQYbYS6WxSbIA&state=xyz789
```

Nothing different here from a non-PKCE flow — this is exactly the step an attacker targets in File 2's interception techniques, and the code is exactly as interceptable as before. PKCE doesn't try to stop interception of the code itself; it stops the intercepted code from being useful.

**Step 5 — Token exchange includes the original verifier**

```
POST /token HTTP/1.1
Host: oauth-authorization-server.com
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&code=SplxlOBeZQQYbYS6WxSbIA&redirect_uri=https://client-app.com/callback&client_id=a1b2c3&code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk
```

**Step 6 — The authorization server verifies**

The server recomputes `BASE64URL-ENCODE(SHA256(code_verifier))` from the submitted `code_verifier` and checks it matches the `code_challenge` it stored back in Step 3. Only if they match does it issue a token.

**Why this actually stops the File 2 attack:** an attacker who intercepts the code in Step 4 (via any redirect_uri manipulation, Referer leak, or otherwise) does **not** have the `code_verifier` — it was never transmitted anywhere and never left the legitimate client's memory. When the attacker tries to redeem the stolen code at the legitimate client application's real callback endpoint (the "replay the code" step that made File 2's attacks work), the legitimate client's backend will generate a token-exchange request using **its own stored verifier for a different, legitimate flow** — which won't match the challenge associated with the stolen code, because that challenge was generated by whatever malicious or unrelated flow the attacker used to obtain the code in the first place. The exchange fails.

## 2. Misconfiguration 1: PKCE Verified at Authorization, Not at Token Exchange

**The flaw:** some implementations check that `code_challenge` and `code_challenge_method` are *present and well-formed* at the authorization step, but never actually store the challenge against the issued code, or never actually perform the SHA-256 comparison at Step 6 — they accept whatever `code_verifier` is submitted without validating it against anything.

**Why this happens in practice:** PKCE is sometimes bolted on to satisfy a compliance checklist or a client library's default behavior without the corresponding server-side verification logic being fully wired up — the parameters are accepted and stored in the request log, but the actual cryptographic check at redemption time is missing.

**How to detect it:** intercept a legitimate flow, capture a valid authorization code, and attempt the token exchange with a `code_verifier` that does **not** correspond to the original `code_challenge` — supply a completely different random string, or omit `code_verifier` entirely if it's not marked as strictly required:

```
POST /token HTTP/1.1
Host: oauth-authorization-server.com

grant_type=authorization_code&code=SplxlOBeZQQYbYS6WxSbIA&redirect_uri=https://client-app.com/callback&client_id=a1b2c3&code_verifier=completely_wrong_value_here
```

If this succeeds and returns a valid token, PKCE is present in name only — the entire File 2 attack surface remains fully exploitable, because an attacker replaying a stolen code never needs a correct verifier at all.

## 3. Misconfiguration 2: code_challenge_method=plain

The PKCE spec allows two transformation methods: `S256` (SHA-256, the mechanism above) and `plain`, where `code_challenge` is simply set equal to `code_verifier` with no hashing at all — included in the spec mainly for constrained clients that can't perform SHA-256 computation, not intended as an equally secure alternative.

**Why plain defeats the purpose:** if `code_challenge_method=plain`, the value that travels in the open in the authorization request (`code_challenge`) **is** the verifier. Anyone who intercepts the authorization request — the exact request an attacker is already constructing in every File 2 technique — has the verifier as well as everything else, and can submit it directly at token exchange:

```
GET /auth?...&code_challenge=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk&code_challenge_method=plain
```

```
POST /token HTTP/1.1
...
code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk
```

Same value, sent twice, with no cryptographic transformation between them. This provides essentially zero protection against code interception — it satisfies "PKCE is technically present" for compliance purposes while providing none of the actual security property.

**How to detect it:** check `code_challenge_method` in every authorization request you observe. If it's `plain`, or if the parameter is omitted and the server silently defaults to accepting a plain-text match, flag this as a finding — even without demonstrating a full exploit chain, "PKCE downgrade to plain" is a legitimate and reportable weakness, since it removes the defense-in-depth PKCE is supposed to provide against every technique in File 2.

**A related downgrade attack worth testing:** even if the client always sends `S256`, check whether the **authorization server** actually enforces that method, or whether it will silently accept a request where an attacker changes `code_challenge_method` to `plain` and submits the SHA-256 digest itself as if it were a plain-text verifier — some poorly implemented servers don't strictly validate that the submitted method matches what's actually required by policy, allowing an attacker to force a downgrade unilaterally within a single request, without needing the legitimate client to ever use `plain` itself.

## 4. Misconfiguration 3: PKCE Only Enforced for Public Clients

**The flaw:** some authorization servers make PKCE mandatory only for clients registered without a `client_secret` (public clients), and treat it as fully optional for confidential clients that do have a secret — reasoning that the `client_secret` check at token exchange already provides equivalent protection.

**Why this reasoning is incomplete:** the `client_secret` check confirms the token-exchange request is coming from a legitimate instance of the *client application's backend* — it says nothing about whether the *specific authorization code* being redeemed actually belongs to the flow that's redeeming it. This is precisely the gap File 2 exploits: the attacker doesn't need the client secret at all, because they simply hand the stolen code to the legitimate backend and let it redeem the code with its own valid secret on the attacker's behalf. A `client_secret` check and a PKCE check protect against two different things — the secret proves "this is really the registered application," while PKCE proves "this code belongs to a flow that this specific instance of the application actually initiated." Making PKCE optional for confidential clients leaves the exact File 2 attack chain open even on applications that otherwise look well-secured.

**How to detect it:** if you're testing a confidential client (one you can confirm uses a `client_secret`), check whether the authorization request is accepted **without** `code_challenge`/`code_challenge_method` at all. If the flow completes normally with no PKCE parameters present, and you can independently confirm (via File 2's techniques) that the code can be intercepted, the application has no protection against code replay beyond redirect_uri validation alone.

## 5. Misconfiguration 4: Verifier Not Bound to a Single Code / Reusable Across Flows

**The flaw:** a subtler bug where the authorization server correctly performs the SHA-256 check, but stores and validates the challenge in a way that isn't strictly scoped to the single authorization code it was issued alongside — for example, keying the stored challenge by `client_id` alone, or by a session identifier that persists across multiple authorization attempts, rather than by the specific `code` value.

**Why this matters:** if an attacker can trigger the client application to initiate *any* authorization request (even one they can't complete or intercept the result of, such as forcing the victim's browser to start a flow), and the resulting challenge gets stored in a way that's reachable by a **different** code the attacker separately controls, the one-to-one binding PKCE is supposed to guarantee between a specific code and a specific verifier breaks down. This is a less common finding than Misconfigurations 1–3 and typically requires careful observation of exactly how the authorization server's session/state management ties challenges to codes — worth checking specifically by attempting to redeem a legitimately-stolen code (via a File 2 technique) using a verifier you generated yourself for an entirely separate, unrelated flow of your own against the same client_id.

## 6. Testing Methodology

1. Identify whether the target uses PKCE at all — look for `code_challenge` and `code_challenge_method` in the authorization request.
2. If absent entirely on a public client (SPA, mobile app, CLI), that's already a finding — File 2's interception techniques apply with no additional mitigation.
3. If present, note the `code_challenge_method` value. `plain` is an immediate flag per Section 3.
4. Attempt token exchange with a deliberately incorrect `code_verifier` (Section 2's test) — if it succeeds, PKCE isn't actually enforced server-side regardless of what the client sends.
5. If testing a confidential client, check whether omitting PKCE parameters entirely from the authorization request is still accepted (Section 4).
6. Combine with File 2: once you've confirmed a redirect_uri or Referer-based technique that successfully steals a code, always additionally attempt to redeem that stolen code through the normal callback flow, and observe whether the exchange fails due to a PKCE mismatch (correctly implemented) or succeeds anyway (bypassed via one of the misconfigurations above) — this combined test is the real-world proof that determines whether a redirect_uri finding escalates all the way to full account takeover or is capped at a lower-severity "code interception, but PKCE prevents redemption" finding.

## Real-World Industry Note

PKCE adoption has increased sharply since OAuth 2.1 effectively made it the default recommendation for all client types, which means testers now encounter it far more often than a few years ago — but adoption of the *parameter* has outpaced correct server-side *enforcement* in a lot of production systems, particularly ones built on older OAuth libraries that added PKCE support as a bolt-on feature rather than a core part of the token-exchange logic. In practice, seeing `code_challenge` in an authorization request should prompt a test, not an assumption of safety — Misconfiguration 1 (verified at the wrong step) is common enough in real assessments that "PKCE is present" and "PKCE is enforced" should always be treated as two separate findings to confirm independently.
