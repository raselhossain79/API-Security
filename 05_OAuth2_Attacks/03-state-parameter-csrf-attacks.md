# state Parameter Bypass: CSRF on OAuth Flows

## How This Differs From File 2

File 2 covered attacks where the *attacker* steals a code or token that belongs to a real, victim-initiated OAuth flow. This file covers the opposite direction: the attacker initiates the OAuth flow themselves — using their own account with the OAuth provider — and tricks the victim's browser into completing it, so the *victim ends up authenticated as, or linked to, the attacker*. This is a classic CSRF pattern applied specifically to OAuth's callback step, and it's a completely different impact model that's easy to conflate with redirect_uri attacks if you don't keep the direction of compromise straight.

Prerequisite: File 1's breakdown of the `state` parameter in the Authorization Code flow, and File 2's explanation of why `state` doesn't stop redirect_uri interception (that's the reverse of what's covered here — read it if you skipped it, since understanding why state fails to stop one class of attack while being essential against another is the key conceptual point of this whole file).

---

## 1. What state Is Actually For

The `state` parameter is generated by the **client application** before redirecting the user to the authorization server, and is expected to come back unchanged in the callback:

```
GET /auth?client_id=a1b2c3&redirect_uri=https://client-app.com/callback&response_type=code&scope=openid%20profile&state=hr48fjeo3n HTTP/1.1
Host: oauth-authorization-server.com
```

```
HTTP/1.1 302 Found
Location: https://client-app.com/callback?code=SplxlOBeZQQYbYS6WxSbIA&state=hr48fjeo3n
```

**The correct mechanism:** before redirecting the user, the client generates an unguessable, session-bound value (commonly a hash tied to the user's current session, or a cryptographically random token stored server-side or in a signed cookie against that session) and stores it. When the callback arrives, the client checks that the `state` value in the callback matches the one it stored **for that specific browser session** — not just that a `state` value is present, and not just that it matches *some* value on record, but that it matches the value generated *for the session currently completing the callback*.

This makes `state` functionally identical to a classic anti-CSRF token, just applied to the OAuth callback endpoint instead of a form submission. That's the correct mental model: **state is CSRF protection for the OAuth flow, nothing more.** It says nothing about who owns the authorization code — it only proves that the browser completing the callback is the same browser that initiated the request.

## 2. Why Missing or Broken state Enables Account-Linking and Login CSRF

Consider a client application that lets users either log in with a password or link a social media account via OAuth for one-click future logins. If `state` is missing or not properly checked, an attacker can:

1. Start the OAuth flow using **their own** social media account.
2. Intercept the resulting callback request (`GET /oauth-linking?code=ATTACKER_CODE`) before it completes, or simply construct it directly from the known flow.
3. Deliver that exact callback URL to the victim — e.g. via an auto-loading iframe on attacker infrastructure.
4. The victim's browser, while authenticated to the client application, loads the URL. Because the client-side session context (the victim's logged-in cookie on the client app) is what determines *whose account* gets the social profile linked, and because there's no `state` value tying this callback back to a specific link-initiation request, the client application completes the flow and attaches the **attacker's** social media identity to the **victim's** account on the client application.
5. The attacker can now log in via "Log in with social media," using their own social credentials, and land directly in the victim's account.

This is the exact mechanism behind PortSwigger's **"Forced OAuth profile linking"** lab, and it's worth internalizing why this is so severe: the attacker never needs to steal any victim credential, cookie, or token. They only need the victim's browser to make one GET request. The account takeover happens entirely through account-linking logic that trusted the wrong signal (an active victim session) instead of the right one (a matching state value proving this callback belongs to a request the victim's session actually initiated).

### Flag-by-flag breakdown of the malicious delivery

```html
<iframe src="https://client-app.com/oauth-linking?code=ATTACKER_STOLEN_CODE"></iframe>
```

- `oauth-linking` — the client application's own callback endpoint for the *account-linking* feature specifically (distinct from the normal login callback). This distinction matters: some applications correctly protect the login callback with `state` but forget to apply the same protection to a secondary linking feature added later, which is a very realistic real-world gap.
- `code=ATTACKER_STOLEN_CODE` — this is a **valid, unused** authorization code belonging to the attacker's own OAuth-provider account. The attacker obtains it by starting the linking flow themselves, intercepting the resulting callback request in Burp *before* it's actually sent to the client application, and copying the code out — the code must still be unused and unexpired when delivered to the victim, so this has to happen shortly before delivery.
- No `state` parameter at all in this request — or, in a weaker but still exploitable variant, a `state` parameter that's present but is never actually compared against a session-bound stored value by the client's backend.

## 3. Login CSRF: A Narrower But Still Real Impact

If the client application only supports login via OAuth (no separate password-based login, no linking feature to hijack), missing `state` is less severe but still exploitable as classic **login CSRF**: an attacker forces the victim's browser to complete an OAuth login flow using the *attacker's own* account, logging the victim into the attacker's account on the client application without the victim realizing it.

This sounds low-impact at first (why would being logged into someone else's account harm the victim?) but it has a well-documented real-world consequence: if the victim doesn't notice the account switch and proceeds to enter sensitive data — payment details, personal information, uploaded documents — into what they believe is their own account, that data is now sitting in an account the attacker controls. This is the mechanism PortSwigger's own topic material calls out explicitly as the reason `state` still matters even on OAuth-only login systems, and it's a genuinely underrated finding class in bug bounty triage because reviewers sometimes dismiss "attacker can log victim into attacker's account" as low severity without considering the data-exfiltration-via-confusion angle.

## 4. What "Broken" state Looks Like in Practice

Not every `state` implementation that exists is actually protective. Test each of the following on any target:

### 4.1 Missing entirely

The most obvious case — no `state` parameter is sent in the authorization request at all. Straightforward to spot in Burp's proxy history.

### 4.2 Present but never validated

`state` is generated and sent, and comes back in the callback, but the backend never actually compares it to anything — it's read and silently ignored. Test this by simply omitting `state` from a callback request you construct yourself, or by substituting an arbitrary value. If the flow completes successfully regardless of what `state` you send, validation isn't happening even though the parameter exists.

### 4.3 Predictable or low-entropy values

If `state` is derived from something guessable — an incrementing counter, a timestamp, a hash of publicly known data (e.g. `md5(username)` when the username is public) — an attacker can construct a valid-looking `state` value without ever having observed a real one. Always inspect the actual value: is it long, random-looking, and different every time you initiate the flow? Or does it look structured, short, or repeat under any observable pattern?

### 4.4 Not bound to the session that generated it

This is the subtlest and most commonly missed variant. `state` might be cryptographically strong and *is* checked against a stored value — but the stored value is looked up by some identifier the attacker can also control or predict (e.g. looked up by `client_id` + `code` rather than by the victim's actual session), rather than being strictly tied to the specific browser session that initiated the request. In this case an attacker can generate a completely valid `state`/code pair from their **own** browser, and it will still validate successfully when replayed from the victim's browser, because the check never actually confirmed *whose* session the state value belongs to.

This is exactly the caveat noted in File 2 and worth restating precisely here since it's the crux of the whole state-parameter topic: **an attacker can always generate their own valid state value by simply starting the flow themselves.** State only protects against forgery — it does nothing to stop an attacker from using a value that's entirely legitimate but was generated for the *wrong* purpose (i.e., their own account-linking flow, delivered to someone else's browser). This is why state alone doesn't stop the redirect_uri interception attacks in File 2, but *does* stop the account-linking CSRF in this file — the difference is what the attacker is trying to achieve, not the strength of the token itself.

## 5. Testing Methodology

1. Proxy a complete, legitimate OAuth flow through Burp and locate the `state` value in both the authorization request and the callback.
2. Send the callback request to Repeater. Remove `state` entirely and resend — does the flow still complete?
3. If present, substitute a clearly invalid value (`state=aaaa`) — does it still complete?
4. If a valid-looking `state` is required, initiate the flow twice from the same browser/session and compare the two `state` values for any structural pattern (sequential, timestamp-based, derivable).
5. If you have two separate accounts/sessions available (as most PortSwigger labs and many real targets provide via test accounts), initiate the flow from Account A, capture the resulting code+state pair, and attempt to replay that exact pair from Account B's browser session before the code expires. If it succeeds, state is not session-bound — this is the account-linking CSRF primitive described in Section 2.
6. If the client application supports account linking specifically (not just login), always test that endpoint's CSRF protection separately from the main login callback — as noted in Section 2, it's a common real-world gap that the linking flow is unprotected even when the primary login flow is not.

## Real-World Industry Note

In production bug bounty programs, `state`-related findings are frequently under-reported relative to their actual prevalence, because testers fixate on the more visually dramatic redirect_uri interception attacks from File 2 and skip the comparatively unglamorous step of testing whether `state` is session-bound. In practice, checking `state` behavior takes about five extra minutes once you already have two test accounts proxying through Burp, and it's consistently worth those five minutes — account-linking CSRF findings routinely get rated Critical because the resulting impact (full account takeover with zero credential theft) is identical to far more complex attack chains.
