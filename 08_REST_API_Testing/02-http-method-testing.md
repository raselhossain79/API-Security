# REST API Testing Methodology — Part 2: HTTP Method Testing (Method Abuse)

## Series Position

This file assumes Part 1 is complete: you have an endpoint matrix, a populated Burp site map, and a recorded baseline for every documented endpoint+method+role combination. This file covers the systematic process of testing **every** HTTP method against **every** endpoint, regardless of what the spec documents — and explains precisely what each result tells you.

---

## Why Method Abuse Testing Matters

REST APIs are built on frameworks (Express, Spring, Django REST Framework, ASP.NET Web API, Rails, FastAPI, etc.) that route requests by method + path. Developers explicitly implement the methods they intend to support (e.g., `GET /users/{id}` and `PUT /users/{id}`), but framework routing behavior and middleware/auth-check placement frequently create gaps:

- A framework may route `DELETE /users/{id}` to the *same handler* as `PUT` if the developer used a wildcard/catch-all method binding, even though `DELETE` was never intended to be exposed.
- Authorization middleware is sometimes applied per-route-per-method inconsistently — e.g., `PUT` is protected by a role check, but `PATCH` on the identical path was added later and the developer forgot to apply the same middleware.
- `OPTIONS` and `HEAD` are often auto-generated by frameworks and left completely unauthenticated by default, sometimes leaking information (allowed methods, sometimes even more) that the documented methods don't.

**Real-world framing:** this exact pattern — an endpoint properly protected on the documented method but wide open on an adjacent undocumented method — is one of the most consistently reported API findings in bug bounty programs, precisely because most testers only exercise the methods shown in the app's normal traffic. It's classified under OWASP API1:2023 (Broken Object Level Authorization) or API5:2023 (Broken Function Level Authorization) depending on what the exposed method allows.

## Step 1: The Method Matrix

For **every** endpoint in your Part 1 matrix, test all of the following methods, not just the documented one:

```
GET, POST, PUT, PATCH, DELETE, OPTIONS, HEAD, TRACE, CONNECT
```

In practice `TRACE` and `CONNECT` are rarely relevant for REST API application-layer testing (they're more relevant to web server-level misconfiguration) — include them only if the target is a bare origin server rather than one behind a modern reverse proxy/CDN, since most CDNs strip or reject them by default. Focus effort on the first seven.

### 1.1 — Constructing the Test Set in Burp Repeater

Take a baselined request (from Part 1, Step 4) and duplicate it once per method using **Repeater's "duplicate tab"** feature, changing only the method line and, where relevant, moving/removing the body.

Example baseline request (documented method — `GET`):

```
GET /api/v2/users/482/profile HTTP/1.1
Host: api.target.com
Authorization: Bearer eyJhbGciOi...
Accept: application/json
```

Now systematically change **only the method line**, keeping headers and auth token identical, and observe:

```
PUT /api/v2/users/482/profile HTTP/1.1
Host: api.target.com
Authorization: Bearer eyJhbGciOi...
Accept: application/json
Content-Type: application/json

{"role":"admin"}
```

**What this specific test checks:** whether the framework has a `PUT` handler bound to this exact path that was never documented, and — critically — whether it enforces the same authorization the `GET` endpoint enforces. The body (`{"role":"admin"}`) is a deliberate probe for mass assignment on top of the method-abuse check — if the server accepts this and returns 200, you've potentially found both an undocumented method *and* a privilege escalation vector in a single test, which is common enough that it's worth always including a plausible sensitive-field body when testing `PUT`/`PATCH`/`POST` on a discovered endpoint.

### 1.2 — Full Method Sweep Template

Run this exact sequence against every endpoint in your matrix:

| # | Method | Body sent | What it specifically tests |
|---|---|---|---|
| 1 | `GET` | none | Baseline (already recorded in Part 1) |
| 2 | `POST` | Same shape as documented create-body, or empty `{}` if unknown | Whether an undocumented create handler exists on a path meant only for retrieval/update |
| 3 | `PUT` | Full valid object matching the resource's documented schema | Whether unauthenticated/under-authorized full-replace is possible |
| 4 | `PATCH` | Single-field partial body, e.g. `{"email":"test@test.com"}` | Whether partial update bypasses checks that full `PUT` enforces (common — some frameworks validate `PUT` bodies against a schema but skip schema validation for `PATCH` because it's "partial") |
| 5 | `DELETE` | none | Whether deletion is possible without the authorization the UI implies is required |
| 6 | `OPTIONS` | none | What methods the server *claims* to support via `Allow` header — compare this claim against what you actually find works |
| 7 | `HEAD` | none | Whether `HEAD` returns headers identical to `GET` (it should) or leaks different information (some frameworks skip auth middleware for `HEAD` because "it's just headers") |

### 1.3 — Reading the OPTIONS Response Correctly

Send `OPTIONS` first, before manually sweeping the rest, since it can shortcut your work — but never trust it as ground truth:

```
OPTIONS /api/v2/users/482/profile HTTP/1.1
Host: api.target.com
Authorization: Bearer eyJhbGciOi...
```

Expected response header to inspect:

```
Allow: GET, PUT, HEAD, OPTIONS
```

**What this tells you, and what it doesn't:**
- This is the framework's own declaration of which methods are routed for this path. Treat it as a *hint*, not a guarantee — it tells you what's *routed*, not what's *authorized correctly*. A method appearing in `Allow` still needs the full auth-differential test (Part 5) to confirm it's properly protected.
- Conversely, a method's *absence* from `Allow` does not guarantee it's unroutable — some frameworks compute the `Allow` header from route declarations that don't perfectly match runtime behavior (e.g., a catch-all route handler that accepts any method but wasn't registered in a way that populates `Allow` correctly). This is why you still manually test methods not listed, rather than skipping them.

## Step 2: Interpreting Results Correctly

This is the part testers get wrong most often — a status code alone is not enough to conclude anything. You must compare the *full response* (status + body + headers) against your Part 1 baseline.

| Response pattern | What it actually means |
|---|---|
| `404 Not Found`, identical body/size to a known-nonexistent-path baseline | Method is genuinely not routed. Low priority. |
| `405 Method Not Allowed` | Method is explicitly rejected by framework routing. This is the *correct*, safe behavior — confirms the framework is properly restricting methods at the routing layer. |
| `200 OK` (or `201`/`204`) with a response shape matching what a legitimate operation would produce | **Confirmed finding** — the method is live and processed the request. Now check: did it enforce the same authorization as the documented method? (Feeds into Part 5.) |
| `500 Internal Server Error` | Ambiguous — could mean the method reached application code and hit an unhandled case (worth investigating further, sometimes leaks stack traces with framework/library version info) or could be a generic error unrelated to the method itself. Compare against whether `500` is also your baseline behavior for malformed requests generally (established in Part 1) before concluding anything. |
| `401`/`403` | Correctly protected — but note **which** methods return this vs. which return `200`, since inconsistency across methods on the *same path* is itself the finding, even if no single method is individually broken. |

**Critical discipline:** always diff the *full response body*, not just the status code, against your baseline. A `200 OK` on an undocumented `DELETE` that actually returns an empty body vs. one that returns the deleted object's full data are different-severity findings — the latter confirms the object genuinely existed and was processed, while the former needs a follow-up `GET` to confirm the resource is actually gone (i.e., the `DELETE` had a real effect, not just a fake-success response).

## Step 3: Method Override Header Testing

Many frameworks support method override via headers or query parameters, intended for clients (like old browsers or restrictive corporate proxies) that can't send `PUT`/`DELETE` natively. These overrides are frequently left enabled in production and are **not covered by the direct method sweep in Step 1**, because they operate on top of a `POST` request rather than changing the request line itself.

Test each of these on top of a plain `POST` to the same path:

```
POST /api/v2/users/482/profile HTTP/1.1
Host: api.target.com
Authorization: Bearer eyJhbGciOi...
X-HTTP-Method-Override: DELETE
Content-Type: application/json

{}
```

**What this specifically tests:** whether the framework has method-override middleware enabled (common in Express via `method-override` package, and in some Spring configurations) that translates this header into an internal `DELETE` — and whether authorization middleware is applied *before or after* that translation. If auth middleware checks the literal HTTP method from the request line (`POST`, which might be allowed for this authenticated user) before the override middleware rewrites it to `DELETE`, you get a `DELETE` executed under an authorization check meant for `POST` — a real, exploitable ordering bug.

Headers/parameters worth testing, in order of real-world prevalence:

```
X-HTTP-Method-Override: DELETE
X-HTTP-Method: DELETE
X-Method-Override: DELETE
_method=DELETE          (as a query parameter or form/JSON body field)
```

Repeat this override test for each of `PUT`, `PATCH`, `DELETE` as the override target, since different frameworks honor different override values and some honor all three.

## Step 4: Case Sensitivity and Method String Variations

Some legacy or custom-routed frameworks are inconsistent about method string casing or unusual verbs. Low-yield but essentially free to test once you're already in Repeater:

```
Delete /api/v2/users/482    (mixed case)
DELETE /api/v2/users/482    (uppercase, standard)
```

**What this tests:** whether routing logic does a case-sensitive string match (correct, standard behavior) or a case-insensitive comparison feeding into a *different* code path that has different auth enforcement than the standard-case route. This is a low-probability finding in modern frameworks (most HTTP libraries normalize method casing before routing) but worth a quick pass on custom/older stacks, especially internal or legacy government/enterprise APIs, which show up disproportionately often in real engagements against older systems.

## Step 5: Automating the Sweep at Scale

Manually repeating Steps 1–4 for every endpoint in a large API (50+ endpoints) is impractical by hand. Once you've validated the manual technique on a handful of endpoints and understand what "normal" looks like for this specific target, automate:

### 5.1 — Burp Intruder for Method Sweeping

- Send the baseline request to Intruder.
- Set the **method** as the payload position (not a parameter — the actual HTTP verb on the request line).
- Payload list: `GET,POST,PUT,PATCH,DELETE,OPTIONS,HEAD`.
- Use **Grep Match** on Intruder's options to flag responses containing indicators of success (`"id":`, `"success":true`, or your baseline's known success markers) so you can sort results by likelihood of a live handler rather than reading every response manually.

### 5.2 — Scripting for Full-Matrix Coverage

For full endpoint-matrix coverage (every endpoint × every method × every role), a small script is more reliable than manual Repeater work and produces a consistent, re-runnable audit trail — valuable for report writing and for re-testing after remediation. Structure:

```
for endpoint in endpoint_matrix:
    for method in [GET, POST, PUT, PATCH, DELETE, OPTIONS, HEAD]:
        for role in [unauth, low_priv, high_priv]:
            send_request(method, endpoint, role_token[role])
            record(status_code, response_body_hash, response_length)
compare_all_results_against_baseline()
```

This pseudocode is the actual shape of the differential engine covered in full in **Part 5: Response Differential Analysis** — method sweeping and role differential are the same underlying technique applied along different axes, which is why baselining (Part 1) is a shared prerequisite for both.

## Summary — Method Testing Checklist Per Endpoint

- [ ] All 7 core methods tested (GET/POST/PUT/PATCH/DELETE/OPTIONS/HEAD)
- [ ] `OPTIONS` response `Allow` header recorded and treated as a hint, not ground truth
- [ ] Every non-404/405 result diffed against Part 1 baseline (status + body + headers, not status alone)
- [ ] Method override headers tested on top of `POST` (`X-HTTP-Method-Override`, `X-HTTP-Method`, `X-Method-Override`, `_method`)
- [ ] Case variation spot-checked on at least one representative endpoint per API (skip if framework is a known modern stack that normalizes casing)
- [ ] Every live undocumented method re-tested across all roles (feeds directly into Part 5)
